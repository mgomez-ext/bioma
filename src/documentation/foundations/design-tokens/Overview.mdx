# Design Tokens

Design tokens abstract out visual properties like color, typography, and spacing for consistent and meaningful user interface design across tools and platforms.

## What are Design Tokens?

Design tokens are the **single source of truth** for design decisions in Bioma. They're named entities that store visual design information, enabling teams to build consistent interfaces without hardcoding values.

Instead of using magic numbers and hex codes scattered throughout your codebase, tokens provide semantic, self-documenting references that make your design system scalable and maintainable.

### Without Tokens
```scss
.button {
  background: #E30038;      // âŒ What is this color?
  padding: 12px 24px;       // âŒ Why these values?
  border-radius: 8px;       // âŒ Is this consistent?
}
```

### With Tokens
```scss
.button {
  background: var(--primary-default);  // âœ… Clear purpose
  padding: var(--spacing-3) var(--spacing-6);  // âœ… Consistent scale
  border-radius: var(--radius-md);    // âœ… System value
}
```

### Tier 1: Foundation Tokens

Foundation tokens are **primitive, context-independent values** that form the system's bedrock.

**Characteristics**:
- Contain absolute values (hex colors, pixel dimensions)
- Never reference other tokens
- Platform and theme agnostic
- Change infrequently
- Serve as building blocks for semantic tokens

**Examples**:
```scss
--brand-500: #E30038;        // Bioma red
--neutral-950: #121212;      // Near black
--spacing-4: 16px;           // Base unit Ã— 4
--radius-md: 8px;            // Medium corner
--font-base: 16px;           // Body text size
```

**Naming Pattern**: `{category}-{scale}` or `{category}-{descriptor}`
- Categories: `brand`, `neutral`, `spacing`, `radius`, `font`, `shadow`
- Scale: Numeric (50-950 for colors, 0-16 for spacing)
- Descriptors: `xs`, `sm`, `md`, `lg`, `xl`, `2xl`, etc.

---

### Tier 2: Semantic Tokens

Semantic tokens provide **contextual meaning** by referencing foundation tokens and adapting to themes.

**Characteristics**:
- Reference foundation tokens using `var()`
- Self-documenting names that explain purpose
- Adapt to context (light/dark mode, themes)
- Used directly in components
- More flexible and maintainable than foundation tokens

**Examples**:
```scss
--primary-default: var(--brand-500);     // Primary brand action
--text-primary: var(--neutral-950);      // Primary text color
--bg-default: var(--neutral-white);      // Default background
--border-default: var(--neutral-300);    // Default border
```

**Naming Pattern**: `{purpose}-{variant}`
- Purpose: `text`, `bg`, `border`, `primary`, `success`, `error`, etc.
- Variant: `default`, `hover`, `active`, `subtle`, `strong`, `disabled`

**Why Two Tiers?**

This architecture provides the best of both worlds:
- **Foundation tokens** ensure visual consistency across primitive values
- **Semantic tokens** enable flexible theming and clear intent
- **Components** reference semantic tokens for automatic theme adaptation

**Usage Guidelines**:

- Use `spacing-1` and `spacing-2` for compact components
- Maintain consistent spacing relationships within components

## Dark Mode with Tokens

Semantic tokens automatically adapt to light/dark themes:

```scss
// Light Mode (default)
:root {
  --text-primary: var(--neutral-950);  // Dark text
  --bg-default: var(--neutral-white);   // Light background
  --border-default: var(--neutral-300); // Light border
}

// Dark Mode (override)
[data-theme="dark"] {
  --text-primary: var(--neutral-100);   // Light text
  --bg-default: var(--neutral-950);     // Dark background
  --border-default: var(--neutral-700); // Dark border
}
```

Components using semantic tokens work in both modes **without code changes**!

### âŒ Don't

**Don't hardcode values**:
